7 ## Build Products
8 The product is a header only library, but the project will build a number of unit tests to support this header as well as some examples
9 and generated documentation.
10 It is not necessary to build these unit tests, documentation, or examples to use the header, but all develoipment should ensure that all these unit tests pass.
11 (i.e. all executables triggered by make test should PASS)

13 # Repository
14 ## Workflow and Branches
15  - Never work directly on the master branch - this is the "stable release" and is tightly controlled
16  - The master shall compile and pass all unit tests at all times.
17  - Commits to the master must only be done after code review and from a working dev branch
18  - devs should keep their own branch in sync with the dev branch.
19  - The dev shuold compile and pass all unit tests. Any build breaks should receive top priority.
20  - Only merge from a feature branch that is already synced with the dev branch.
21  - Commits to the dev should usually only be done after code review and from a working feature branch
22  - use the github merge request feature for feature->dev branch merges

24 ## Repository Structure
25 The astrotypes library is envisioned to support languages used in astro development. Each language has its own subdirectory
26 below src. Each language directory will have specifics to the language but will usually contain:
27 
28 cmake : specific files for configuring the build system to the specific language
29 tools : useful things to aid the developer such as a class template generator
30 doc : documentation & doc templates, with the exception of the API doc of the classess.
31 astrotypes : this is the actual src code. It may have a substructure e.g. for c++
32 
33  astrotypes +
34  + <module_name> + In this directory only put header files that you wish exposed to module users
35  + doc Add additional doxygen documentation for tutorials etc here.
36  + examples src code example uses
37  + src All .cpp files for the library and internal headers
38  + detail All .cpp implementations details for template classes that cannot be precompiled
39  + test_utils + Include files for the supporting test library
40  + src Implementation details (.cpp files) for the test library
41  + detail Template implementation details for the test support library
42  + test The unit test directory. Header files for unit tests live here
43  + src Implementation details (.cpp files) for unit tests

49 # The Build System
50 This project uses cmake as is build system. See the notes in the CMakeLists.txt file in the top directory

52 ## Adding a src file/class
53  - Each time you add a new class, in order for it to build you will need to add a reference to it in the appropriate CMakeLists.txt file
54  (either in astrotypes/module_name/CMakeLists.txt or astrotypes/module_name/test_utils/CMakeLists.txt)
55  - place source names in alphabetical order when adding to variables
56  - You can use the class tool (in the tools directory) to autogenerate a suitable class template
57 
58 ## Adding a submodule
59  - To add a submodule use the SUBPACKAGE macro (see doc in cmake/subpackage.cmake).
60  To simplify, the purpose of this macro is to add the submodules contribution to the lib_src_${arch} variables which
61  are used to define the contents of the astrotypes libraries.
62  (n.b ensure the variables defined in the submodule use PARENT_SCOPE qualifier)
63

66  - add a new file cmake file in the cmake directory for that product/feature and include this from cmake/dependencies.cmake

73 # Coding Conventions
74 Coding conventions are used in this project to ensure conformity in style and to help
75 reduce any of the very many pitfalls c++ programming can introduce,
76 Remember that these are just general guidelines. Try and stick to them as much as possible,
77 but there will always be cases where doing it differently is justified. Document these exceptions with comments in the code.
78 
79 ## Namespaces
80 - all classes shall be in the pss::astrotypes::<module_name> or pss::astrotypes::<module_name>::test namespace as appropriate.
81 - avoid using where possible "using namespace ..". Avoid completely in public header files.
82 
83 ## Naming Conventions
84 - class names shall start with a capital letter and be CamelCased
85 - method and functions shall be all lower case, with word seperation with an "_" e.g. do_something()
86 - variable and member names shall be all lower case
87 - all members shall be prefixed with an underscore "_"
88 - getters and setters should be overloads with the same function name e.g.
89 ```
90  // getter
91  ThingyType thingy() const;
92 
93  // setter
94  void thingy(ThingyType const& thingy);
95 ```
96 
97 
98 ## Indendation & CRLF of source files
99 - use 4 spaces instead of tab.
100 - use LF in preference to CRLF
101 - namespaces shall not be indented
102 
103 ## C++ Files
104 Each class shall have its own separate cpp and header file. The name of the file should be the same as the class with the .h or .cpp extension
105 Do not mix two classes together in the same file.
106 An exception to this rule can be made where there is a private Internal Implementation class unique to the
107 class being implemented AND it is not confusing to do so.
108 
109 - All classes must be in the pss, astrotypes namespace
110 - Header files must start with.
111 ```
112 #ifdef PSS_ASTROTYPES_<MODULE_NAME>_<CLASS_NAME>_H
113 #define PSS_ASTROTYPES_<MODULE_NAME>_<CLASS_NAME>_H
114 ... body of header
115 #endif // PSS_ASTROTYPES_<MODULE_NAME>_<CLASS_NAME>_H
116 ```
117  In the tool directory you will find a class template generator that you may find helpful.
118 ## Documentation
119 - Add doxygen style comments for each method and each parameter in every public header file.
120 - Think about adding an example of the use of each method
121 - Document the bounds of any parameter
122 - Document untested use cases/parameters (i.e. where you have not provided an explicit unit test)
123 - Add comments as appropriate in the cpp file and non-public headers.
124 
125 ## Constructors
126 - Never use new inside a constructor parameter list e.g.
127 ```
128 dodgy = MyClass(new BadThingToDo());.
129 ```
130  This could easily lead to a memory leak if the constructor of MyClass should fail.
131 - Do not create long lists of parameters with the same types. Use strong typeing as much as possible to
132  get your compiler to do the debugging for you.
133  e.g. thing(int x_dimension, int y_dimension, int subject_index)
134  could become:
135  thing( Dimension x, Dimension y, SubjectIndex i );
136  or even:
137  thing( DimensionX x, DimensionY y, SubjectIndex i );
138 - align commas in the initialiser list on seperate lines
139  e.g.
140 ```c++
141  MyClass(A a, B b)
142  : _a(a) // indent : by 4 spaces on its own line
143  , _b(b) // align comma under the :
144  {
145  }
146 ```

149 ## Class Declarations
150 - all members shall be prefixed with an underscore "_"
151 - all members shall be made either private or protected as appropriate
152 - indent private, protected, amd public keywords by 4 spaces
153 - always use the override keyword when you inherit from a method
154 - do not mix members, typdefs, or functions together, but keep them in their own public:, protected:, or private: blocks.
155 - all public methods and typedef should be documented
156 - header shall be ordered as much as possible as follows, each type in their own public/private/protected block:
157  e.g.
158 ```c++
159  class MyClass
160  {
161  public:
162  // typdefs
163 
164  public:
165  // public methods before protected
166 
167  protected:
168  // protected methods before private
169 
170  private:
171  // private methods
172 
173  protected:
174  // protected members
175 
176  private:
177  // private members
178  };
179 ```

181 ## Exceptions
182 - Prefer exceptions, but use return codes where it is more appropriate
183 - No exceptions to be propagated outside of its thread - think about passing it along with a std::exception_ptr.
184 - Use std::exceptions as appropriate, or panda::Error if you want to add messages with runtime information.
185 - If you need your own exceptions for some reason, try and avoid any memory allocation of the exception. i.e follow the std::exception model
186  It should also be OK to inherit from panda::Error.

188 ## Unit Tests
189 - gtest shall be used as the unit test framework
190 - each class should have its own unit test suite which shall be in files <CLASS_NAME>Test.cpp and <CLASS_NAME>Test.h in the unit test directory
191 - any common functionality should be considered for inclusion in the test utility library, and provded with appropriate headers and documentation
192 - all unit tests should be verified to pass before making a merge reuest
193 - unit test names shall follow the function naming convention of the c++ irrespective of whether it is implemented as a class or not
194 - unit test groups shall be CamelCased
195  e.g
196 ```
197 TEST_F(MyTestGroup, test_something_really_well)
198 ```
199 - to run all the tests
200 ```
201 make test
202 ```
203 - to run all the tests and generate a test coverage report (only in 'coverage' type builds)
204 ```
205 make coverage_report_test
206 ```

208 ## Using CUDA
209 If CUDA is available and it has been explicitly activated in the build system, the ENABLE_CUDA flag will be set.
210 Use this flag to ensure your code compiles and your unit tests run with or without CUDA availability.

213 e.g.
214 
215 cuda_subpackage_compile(${lib_src_cuda} OPTIONS "-arch compute_35")

217 ## Extending std classes
218 While this style guide requests CamelCase typedefs, there are cases where the standard convention is to use lower_case
219 names for typedefs. This is true in the case of classes that extend or can be used in place of classes from the
220 standard library (or as in the case of thrust, boost, etc. libraries that follow conventions from the standard library) that conform to specific concepts. Places where this is done should be clearly documented.
